# 数据准确性保障机制

## 🎯 核心原则

**零容忍幻觉，100%准确可靠**

---

## 🚫 严禁行为（杜绝幻觉）

### 1. 严禁臆造数据
- ❌ 禁止填充缺失数据（如用平均值、插值法填补）
- ❌ 禁止推算历史价格
- ❌ 禁止修改API返回的原始数据
- ✅ **只能原样存储API返回的数据**

### 2. 严禁修改标的清单
- ❌ 禁止添加未在 prompt 中定义的标的
- ❌ 禁止删除 prompt 中定义的标的
- ❌ 禁止修改标的代码或名称
- ✅ **严格按 `kb/market_constants.py` 中的 TARGET_STOCKS 执行（25只）**

### 3. 严禁更改数据源
- ❌ 禁止使用未经验证的第三方数据源
- ❌ 禁止混用不同数据源的数据
- ✅ A股/ETF 只能用 baostock
- ✅ 美股只能用 Yahoo Finance API

### 4. 严禁篡改时间
- ❌ 禁止修改交易日期
- ❌ 禁止调整时区（美股必须用 US/Eastern）
- ✅ 严格按 API 返回的日期存储

---

## ✅ 5层数据验证机制

### Level 1: API响应验证
```python
# 在数据获取时自动执行
- HTTP状态码检查（200 OK）
- JSON格式验证
- 必填字段检查（open, high, low, close, volume）
```

**失败处理**: 记录错误，重试1次，仍失败则标记该标的为失败

---

### Level 2: 数据质量验证
```python
# validate_data_quality() 函数执行

1. 数值范围检查
   - 收盘价 > 0
   - 成交量 >= 0
   
2. OHLC逻辑检查
   - high >= low
   - high >= close
   - low <= open
   
3. 日期格式检查
   - 必须符合 YYYY-MM-DD 格式
```

**失败处理**: 记录警告，存储数据但标记异常

---

### Level 3: 业务逻辑验证
```python
# validate_data_quality() 函数执行

1. 数据量合理性
   - 至少应有60天数据
   
2. 日期连续性
   - 缺口 < 5个交易日
   
3. 涨跌幅合理性
   - A股：单日涨跌停限制10%
   - 美股：通常 < 20%（超过则警告）
```

**失败处理**: 记录警告，建议人工检查

---

### Level 4: 数据库完整性验证
```python
# 入库后自动检查

1. 主键唯一性
   - SELECT symbol, trade_date, COUNT(*) 
     FROM stock_daily_quotes 
     GROUP BY symbol, trade_date 
     HAVING COUNT(*) > 1
   - 应该返回空结果
   
2. 外键约束
   - market 必须是 "A股" / "ETF" / "美股"
   - industry_chain 必须在定义范围内
   
3. 索引有效性
   - idx_quotes_symbol_date 索引正常
```

**失败处理**: **立即停止**，回滚事务，修复后重新导入

---

### Level 5: 跨源一致性验证（可选）
```python
# 对于关键标的，对比多个数据源

1. Yahoo API vs yfinance（美股）
   - 收盘价差异 < 1%
   
2. baostock vs 交易所官网（A股，手动抽检）
   - 随机抽取5个交易日对比
```

**失败处理**: 记录差异，人工确认哪个数据源更准确

## 🔍 数据溯源机制

所有数据必须可追溯：

### 1. 来源明确
- 每条数据都知道来自哪个API
- 记录在 `market` 字段（A股/ETF/美股）

### 2. 时间戳记录
- `created_at`: 数据入库时间
- `trade_date`: 交易日期

### 3. 版本控制
- 使用 `INSERT OR REPLACE`，保留最新数据
- 如需保留历史版本，可创建 `stock_daily_quotes_history` 表

### 4. 审计日志
- 每次导入记录：
  - 时间
  - 标的数
  - 成功/失败情况
  - 验证结果

## 📝 实际案例

### 案例1: API返回异常数据
**场景**: Yahoo API返回某天的最高价 < 最低价

**处理**:
1. Level 2验证检测到错误
2. 记录错误：`存在最高价<最低价的异常数据`
3. **拒绝入库**，该标的数据保持原状
4. 日志记录：`ERROR: NVDA: 数据验证失败 - 存在最高价<最低价的异常数据`
5. 建议：等待下次API修复或手动修正

### 案例2: 数据缺口
**场景**: 某标的缺失3天的数据

**处理**:
1. Level 3验证检测到缺口
2. 记录警告：`存在1个数据缺口（>3天）`
3. **允许入库**（缺口在可接受范围内）
4. 日志记录：`WARNING: AMD: 存在1个数据缺口（>3天）`
5. 建议：检查数据源或网络问题

### 案例3: 涨跌幅异常
**场景**: 某美股单日涨幅45%

**处理**:
1. Level 3验证检测到异常
2. 记录警告：`存在1天涨跌幅超过20%`
3. **允许入库**（美股无涨跌停限制，可能是真实波动）
4. 日志记录：`WARNING: TSLA: 存在1天涨跌幅超过20%`
5. 建议：人工确认是否为真实行情（如财报发布日）

## 🎯 总结

### 核心保障
1. ✅ **严禁幻觉**：不臆造、不修改、不推算
2. ✅ **严格验证**：5层验证机制，层层把关
3. ✅ **自动拒绝**：验证失败的数据拒绝入库
4. ✅ **完整溯源**：每条数据都可追溯到源头
5. ✅ **详细日志**：记录所有警告和错误

### 效果
- **数据准确性**: 100%（通过验证的数据）
- **数据完整性**: >95%（25个标的中至少24个有数据）
- **数据及时性**: T+1（每天更新最新数据）
- **零幻觉**: 所有数据都来自权威API，未经任何修改
